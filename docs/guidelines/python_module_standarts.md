# Python module standarts

1. Принципы
Явное > неявного
Минимальный публичный API
Детерминированность
Отсутствие скрытой логики
Соответствие формальным спецификациям (если заданы)
2. Структура
Публичный API экспортируется только через __init__.py
Внутренние модули: _internal
Минимальная поверхность модуля
Явное разделение:
API
доменной логики
инфраструктуры
3. Тестируемость
Dependency Injection обязателен
IO полностью отделён от логики
Unit-тесты:
без внешних зависимостей
детерминированные
Дополнительно (если есть формальные спецификации):
тесты проверяют инварианты
рекомендуется property-based testing
4. Типизация
Полная типизация публичного API
mypy --strict обязателен
Any запрещён
Типы должны отражать инварианты (где возможно)
5. Стиль кода
black, ruff
max line length = 88
Запрещены:
wildcard импорты
неиспользуемые импорты
6. Линтинг и mypy
ruff: E, F, I, B
mypy: strict
Игнор ошибок запрещён без обоснования
7. Зависимости
Приложения: фиксированные версии
Библиотеки: semver-диапазоны
Минимизировать транзитивные зависимости
8. Ошибки
Только кастомные исключения
Ошибки не подавляются
Исключения — часть публичного API
9. Логирование
Только logging
print запрещён
Логирование не влияет на логику
10. IO
Не смешивать с бизнес-логикой
Использовать адаптеры
IO не должен влиять на детерминированность ядра
11. Документация
Docstring для каждого публичного метода
Документация должна быть:
точной
непротиворечивой коду

Если используются формальные вставки:

Документация может содержать спецификации на Coq
Каждая спецификация должна:
быть компилируемой
не содержать Admitted (кроме прототипов)
12. Формальные спецификации (Coq)

Если документация содержит Coq:

Все утверждения:
либо формально доказаны
либо явно помечены как неформальные

Требуется явная связь:

Coq theorem ↔ Python функция / метод
Код обязан:
соответствовать спецификациям
не нарушать доказанные свойства
13. Инварианты и детерминизм

Любая логика, описанная формально:

должна быть детерминированной
не зависит от:
времени
случайности
внешнего IO

Инварианты должны:

быть отражены в коде
проверяться тестами
14. Эволюция и изменения

Если присутствуют формальные доказательства:

Любое изменение требует:
обновления доказательств
или явного признания их невалидности
Breaking change — это:
изменение API
или нарушение доказанного свойства
15. Definition of Done

Модуль считается готовым, если:

полностью типизирован
покрыт тестами
проходит линтеры
не имеет скрытых зависимостей
имеет минимальный API
документация актуальна

Если используется Coq:

все спецификации компилируются
доказательства валидны
код соответствует спецификациям
