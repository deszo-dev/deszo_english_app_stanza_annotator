# CLI transform module guidelines

1. Архитектура

CLI:

парсинг аргументов
вызов pipeline

Вся логика:

находится в core
реализует чистую трансформацию

Модель:

input -> transform -> output

Core:

не содержит IO
не зависит от окружения
является кандидатом на формальную спецификацию
2. Контракт CLI

Аргументы:

стабильные
обратно совместимые

Вывод:

stdout - только результат
stderr - только логи и ошибки

Exit codes:

0 - success
1 - expected (data) error
2+ - system error

CLI-контракт:

детерминирован
не изменяет семантику core
3. Контракт трансформации

Основное определение:

output = f(input, config)

Свойства:

без побочных эффектов
без IO
без глобального состояния

Если используется Coq:

функция f должна соответствовать формальной спецификации
поведение должно быть доказуемым
4. Вход / Выход

Поддержка:

stdin
файлов

Формат вывода:

строго определён
полностью воспроизводим

Частичные результаты:

запрещены
или явно специфицированы

CLI не должен:

трансформировать данные неявно
менять структуру без спецификации
5. Детерминизм
одинаковый input -> одинаковый output
нет скрытых зависимостей
фиксированные seed (если применимо)

Запрещено:

зависеть от времени
от случайности (без контроля)
от внешнего состояния

CLI не добавляет недетерминизм поверх core
6. Валидация и ошибки

Валидация:

выполняется до передачи в core

Ошибки:

явные
логируются

Exit codes:

строго соответствуют типу ошибки

Нарушение предусловий:

expected error (1)
7. Логирование (обязательно)

Логируются:

входные параметры
шаги pipeline
результат

Каждый шаг:

start + end

Уровни:

DEBUG / INFO / WARNING / ERROR

Гарантии:

данные не смешиваются с логами
логирование не влияет на результат
8. Debug режим (обязательно)
--debug / -d

Обеспечивает:

полную трассировку
промежуточные состояния

Debug:

не изменяет вычисления
только увеличивает наблюдаемость
9. IO и изоляция

CLI:

работает с IO

Core:

работает только с данными

Все внешние зависимости:

изолированы
вынесены в адаптеры

Формальные гарантии:

применяются только к core
10. Производительность

Поддержка streaming:

для больших данных

Требования:

минимизация IO
контроль памяти

Оптимизации:

не должны нарушать детерминизм
и формальные свойства
11. Конфигурация

Приоритет:

CLI args
ENV
defaults

Разрешение конфигурации:

детерминированное
без скрытых источников
12. Безопасность
не логировать чувствительные данные
валидировать и санитизировать вход
ошибки не раскрывают внутренности системы
13. Чистота
нет print
нет смешивания:
данных
логов

CLI остаётся тонким слоем
14. Формальная спецификация (Coq)

Если используется Coq:

Требования к core:

Спецификация:

компилируется
не содержит Admitted (кроме прототипов)

Явное соответствие:

Coq specification <-> function f

Гарантии:

инварианты доказаны
поведение полностью определено
15. Граница CLI и Core

CLI обязан:

передавать данные без искажений
сохранять инварианты
соблюдать предусловия

CLI не имеет права:

изменять порядок вычислений
добавлять побочные эффекты
менять семантику
16. Инварианты и проверяемость

Если есть Coq:

Инварианты:

должны сохраняться при любом CLI-вызове

CLI обязан:

валидировать вход
не пропускать некорректные данные

Тесты:

проверяют соответствие инвариантам
17. Эволюция

Любое изменение:

требует обновления спецификации
или явного признания её невалидности

Breaking change:

изменение CLI-контракта
или нарушение доказанного свойства
18. Definition of Done

Модуль готов, если:

реализует чистую трансформацию
имеет стабильный CLI контракт
детерминирован
полностью логируем
корректно разделяет stdout/stderr
поддерживает debug
валидирует вход
корректно обрабатывает ошибки
не имеет побочных эффектов

Дополнительно (если используется Coq):

спецификации компилируются
доказательства валидны
core соответствует спецификациям
CLI не нарушает формальные гарантии
